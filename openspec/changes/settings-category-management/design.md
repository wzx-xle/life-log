## Context

当前分类系统：6 个预设分类硬编码在 `src/types/index.ts` 中（`Category` 联合类型），每个有固定标签和颜色。预设分类不可修改。`custom` 分类配合 `Place.customCategory` 字段实现自由文本输入，但没有"自定义分类模板"的概念——用户每次输入都是新的字符串，无法复用。

目标：允许用户创建、管理自己的自定义分类，在店铺表单中与预设分类并列展示，提升分类的灵活性和一致性。

约束：
- 不在构建时引入额外依赖
- 保持对现有 `Category` 联合类型的兼容
- 预设分类不可编辑/删除
- 删除自定义分类不丢失店铺数据

## Goals / Non-Goals

**Goals:**
- 用户能在设置页创建、编辑、删除自定义分类（名称 + 颜色）
- 自定义分类与预设分类在店铺表单中并列展示
- 选择自定义分类时，Place 存储 `category: 'custom'` + `customCategory: <分类名>`
- 自定义分类显示对应颜色标记，在列表/详情/地图中可见
- 数据持久化在 IndexedDB

**Non-Goals:**
- 不支持修改预设分类
- 不支持分类排序/拖拽
- 不改变现有 Place 的索引结构
- 不支持自定义分类图标（仅颜色）
- 不引入嵌套/层级分类

## Decisions

### 1. 存储方案：IndexedDB 新表 `customCategories`

**选型**：在 `LifeLogDB` 中新增 `customCategories` 表，而非 localStorage 或复用现有表。

**理由**：
- IndexedDB 已在项目中使用（Dexie），增加表成本极低
- 数据结构化存储比 localStorage 中 JSON 序列化更可靠
- 与 `places`/`reviews` 一致的管理方式

**替代方案**：
- localStorage：简单但数据结构变更时难迁移，且与项目现有模式不一致
- `places` 表加字段：污染店铺数据模型，语义不清

### 2. 自定义分类与预设分类的关系：独立存储，UI 合并展示

**选型**：预设分类保持硬编码（`Category` 联合类型），自定义分类存储在 `customCategories` 表，UI 层将两者合并为一个列表展示。

**理由**：
- 预设分类逻辑不变，零迁移成本
- Place 数据模型不变：选择自定义分类时仍用 `category: 'custom'` + `customCategory` 字段
- 向后完全兼容，旧数据不受影响

**替代方案**：
- 所有分类统一存入 IndexedDB：需要迁移 Place 表、修改 `category` 字段类型，破坏索引，风险高
- 完全替换预设为自定义：破坏现有筛选/统计逻辑

### 3. 自定义分类与 Place 的关联：弱引用（名称匹配）

**选型**：Place 仅存储 `customCategory` 字符串，不与 `customCategories` 表的外键关联。

**理由**：
- 删除自定义分类时，店铺的 `customCategory` 字段保留原值（降级为普通自定义文本），不丢数据
- 编辑自定义分类名称时，通过名称查找旧值实时更新所有关联店铺（事务性批量更新）
- 实现简单，无需引入外键约束

**替代方案**：
- 用 `customCategories.id` 作为外键：删除分类时需要处理级联，复杂度高

### 4. 自定义分类颜色：预设 10 色 + 自定义色值

**选型**：提供 10 个预设颜色供选择，同时支持用户输入 CSS 色值。

**理由**：
- 预设颜色降低选择成本，与预设分类颜色风格一致
- 保留自定义色值灵活性

## Risks / Trade-offs

- **删除自定义分类后店铺仍保留旧名称**：编辑分类时同步更新所有关联 Place 的 `customCategory` 字段；删除时保留原值 → 风险低
- **自定义分类名称变更后筛选失效风险**：由于 `customCategory` 是字符串，改名后旧名称不再匹配新的筛选条件。可接受——用户主动改名，知道后果。通过批量更新关联 Place 来规避
- **customCategories 表无用户数据时不自动初始化预设**：预设分类保持硬编码，customCategories 表仅存储用户自定义分类，首次使用为空 → 正常行为
- **Dexie schema 升级**：新增表需要 `this.version(2).stores(...)` 升级逻辑 → 确保首次安装和升级场景均覆盖
